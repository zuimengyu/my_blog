第8章：三层架构（小白完整版·企业标配）

8.0 前置核心：为啥要搞三层架构？

大白话： 之前的代码是「一锅炖」：Servlet里又写业务逻辑、又写JDBC、又处理页面跳转，代码乱成一团！
分层就像「餐厅分工」：
📌 服务员（表现层）：只负责和顾客交互（接收请求、返回响应）
📌 厨师（业务层）：只负责做菜（处理业务逻辑，如登录验证）
📌 采购（数据层）：只负责买菜（和数据库交互，DAO）

💡 分层好处：
1. 代码清晰，易维护（改业务逻辑不用动DAO，改DAO不用动Servlet）
2. 易扩展（比如换数据库，只改DAO层）
3. 易分工（前端写页面、后端写Service、数据库写DAO）

8.1 三层架构核心（90%项目都这么分）

三层架构（小白版定义）
层级 全称 大白话职责 对应包名/类
表现层 Presentation Layer 接收请求、返回响应、页面跳转 com.servlet（Servlet）、JSP/HTML
业务层 Business Layer 处理业务逻辑（登录、权限、校验） com.service（Service接口+Impl实现）
数据层 Data Access Layer 只和数据库交互（增删改查） com.dao（DAO接口+Impl实现）

额外补充层（辅助）：
📌 实体层（com.pojo）：封装数据库表数据（User、Product等）
📌 工具层（com.util）：工具类（DruidUtil、编码工具等）

8.3.3 3. 业务层（Service）

复用第7章的UserService（略），补充异常处理（企业必备）：

package com.service.impl;

import com.dao.UserDao;
import com.dao.impl.UserDaoImpl;
import com.pojo.User;
import com.service.UserService;

// 业务层实现类（处理业务逻辑+异常）
public class UserServiceImpl implements UserService {
    private UserDao userDao = new UserDaoImpl();

    @Override
    public boolean login(String username, String password) {
        // 1. 非空校验（业务逻辑）
        if (username == null || username.trim().isEmpty()) {
            throw new RuntimeException("用户名不能为空！");
        }
        if (password == null || password.trim().isEmpty()) {
            throw new RuntimeException("密码不能为空！");
        }

        // 2. 调用DAO查询用户
        User user = userDao.findByUsername(username);
        if (user == null) {
            throw new RuntimeException("用户名不存在！");
        }

        // 3. 密码校验
        if (!user.getPassword().equals(password)) {
            throw new RuntimeException("密码错误！");
        }

        return true;
    }
}

8.3.4 4. 表现层（Servlet）

package com.servlet;

import com.service.UserService;
import com.service.impl.UserServiceImpl;
import jakarta.servlet.annotation.WebServlet;
import jakarta.servlet.http.HttpServlet;
import jakarta.servlet.http.HttpServletRequest;
import jakarta.servlet.http.HttpServletResponse;
import jakarta.servlet.http.HttpSession;
import java.io.IOException;

// 表现层：只负责接收请求、调用Service、返回响应
@WebServlet("/login")
public class LoginServlet extends HttpServlet {
    // 注入Service（表现层只依赖Service，不直接依赖DAO）
    private UserService userService = new UserServiceImpl();

    @Override
    protected void doPost(HttpServletRequest request, HttpServletResponse response) throws IOException {
        // 1. 编码（表现层职责）
        request.setCharacterEncoding("UTF-8");
        response.setContentType("text/html;charset=UTF-8");

        // 2. 获取请求参数（表现层职责）
        String username = request.getParameter("username");
        String password = request.getParameter("password");

        try {
            // 3. 调用业务层（表现层不处理业务逻辑，只调用）
            boolean isLogin = userService.login(username, password);

            // 4. 处理响应（表现层职责）
            if (isLogin) {
                HttpSession session = request.getSession();
                session.setAttribute("loginUser", username);
                response.sendRedirect("/home.jsp"); // 登录成功跳主页
            }
        } catch (RuntimeException e) {
            // 5. 异常处理（表现层职责）
            response.getWriter().write("登录失败：" + e.getMessage());
        }
    }
}

8.4 分层解耦：接口与实现类分离（核心）

大白话： 为什么要写接口（UserDao/UserService），再写实现类（UserDaoImpl/UserServiceImpl）？
👉 比如你想换数据库（MySQL→Oracle），只需要写一个 UserDaoOracleImpl，Service层不用改任何代码！

💡 企业套路：
1. 定义接口：只写方法名，不写实现（规范）
2. 写实现类：实现接口的方法（具体逻辑）
3. 上层依赖接口，不依赖实现类（如Servlet依赖UserService接口，不是UserServiceImpl） 
// 表现层只依赖接口，不依赖实现类（解耦）
private UserService userService = new UserServiceImpl(); 
// 如果要换实现，只改这一行：
// private UserService userService = new UserServiceNewImpl();

  • 🍉 三层架构核心：表现层（Servlet）→ 业务层（Service）→ 数据层（DAO），每层只做自己的事
  • 🔐 包名规范：com.servlet / com.service / com.dao / com.pojo / com.util
  • ⚡ 解耦关键：接口+实现类分离，上层依赖接口不依赖实现
  • 🎯 企业流程：前端请求 → Servlet（收参）→ Service（业务）→ DAO（数据库）→ 原路返回响应
