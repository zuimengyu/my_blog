Nginx 核心前置说明
1. 核心定位：高性能HTTP服务器/反向代理服务器，主打静态资源服务、反向代理、负载均衡、请求转发，占用资源少、并发能力强；
2. 核心优势：处理静态资源性能是Tomcat的10倍以上，可作为Java后端的前置代理，实现动静分离、负载均衡；
3. 配置原则：核心配置文件nginx.conf，采用「块级语法」，分号结尾、花括号包裹，支持include引入子配置；
4. 操作核心：配置修改后先检测语法，再热重载（无需重启，不中断服务），这是生产环境标准操作；
5. 路径说明：Windows配置文件在conf/nginx.conf，Linux在/etc/nginx/nginx.conf或/usr/local/nginx/conf/nginx.conf。

模块1：Nginx 快速安装（Windows+Linux 双环境）

覆盖主流开发/生产环境，步骤极简，无需复杂配置，安装完成即可启动使用，是所有操作的基础。

1.1 Windows 安装（开发环境）

  1. 下载：官网（nginx.org）下载稳定版（Stable version），如nginx-1.24.0；
  2. 解压：将压缩包解压到无中文、无空格的路径，如D:\nginx-1.24.0；
  3. 验证：进入解压目录，双击nginx.exe，浏览器访问http://127.0.0.1，出现「Welcome to nginx!」即安装成功；
  4. 注意：Windows下Nginx为后台进程，需通过命令行启动/停止/重载，不可直接关闭窗口。

Windows 核心目录结构

D:\nginx-1.24.0
├─ conf/ # 配置文件目录（核心：nginx.conf）
├─ html/ # 默认静态资源目录（首页、错误页）
├─ logs/ # 日志目录（access.log访问日志、error.log错误日志）
├─ sbin/ # 执行文件目录（nginx.exe）
└─ temp/ # 临时文件目录

1.2 Linux 安装（生产环境，CentOS/Ubuntu 通用）

推荐使用yum/apt一键安装，自动配置环境变量，无需手动解压，生产环境首选。

CentOS/RHEL 安装

# 1. 安装依赖
yum install -y gcc pcre pcre-devel zlib zlib-devel openssl openssl-devel
# 2. 一键安装Nginx
yum install -y nginx
# 3. 设置开机自启
systemctl enable nginx
# 4. 启动并验证
systemctl start nginx && curl http://127.0.0.1

Ubuntu/Debian 安装

# 1. 更新源
apt update && apt upgrade -y
# 2. 一键安装Nginx
apt install -y nginx
# 3. 开机自启+启动
systemctl enable --now nginx && curl http://127.0.0.1

Linux 核心目录结构

/etc/nginx/ # 主配置目录（nginx.conf、conf.d/子配置）
/usr/share/nginx/html/ # 默认静态资源目录
/var/log/nginx/ # 日志目录（access.log、error.log）
/usr/sbin/nginx # 执行文件
/var/run/nginx.pid # 进程PID文件
安装验证通用方法：浏览器访问http://服务器IP/127.0.0.1，出现Nginx默认欢迎页，即安装成功；若访问失败，检查防火墙（开放80端口）。

模块2：Nginx 核心配置文件（nginx.conf）全解析

Nginx所有功能均通过nginx.conf配置，采用**块级嵌套语法**，从顶层到底层分为「全局块 → events块 → http块 → server块 → location块」，层层递进，**http块可包含多个server块（虚拟主机），server块可包含多个location块（路径匹配）**。

完整核心配置模板（带详细注释，可直接替换原配置）

# ================================= 全局块（全局配置，作用于整个Nginx）=================================
# 工作进程数：推荐设置为「CPU核心数」或「CPU核心数*2」，如CPU4核设为4/8
worker_processes  1;
# 错误日志路径+级别：debug/info/warn/error/crit，生产环境用warn/error
error_log  logs/error.log  warn;
# 进程PID文件路径
pid        logs/nginx.pid;

# ================================= events块（网络连接配置）=================================
events {
    # 单个工作进程的最大并发连接数
    worker_connections  1024;
    # 连接复用模式：epoll（Linux）/kqueue（Mac）/select（Windows），Linux默认epoll，高性能
    use epoll;
    # 允许单个进程同时接受多个新连接
    multi_accept on;
}

# ================================= http块（HTTP核心配置，可包含多个server）=================================
# 所有HTTP/HTTPS相关配置都在http块中，可配置全局参数、反向代理、缓存、日志等
http {
    # 引入MIME类型映射文件（识别文件后缀，如.html/.css/.js）
    include       mime.types;
    # 默认MIME类型：二进制流（若未识别文件后缀，按二进制下载）
    default_type  application/octet-stream;

    # 自定义访问日志格式：$remote_addr(客户端IP) $request(请求行) $status(状态码) $body_bytes_sent(发送字节数) $request_time(请求耗时)
    log_format  main  '$remote_addr - $remote_user [$time_local] "$request" '
                      '$status $body_bytes_sent "$http_referer" '
                      '"$http_user_agent" "$http_x_forwarded_for"';
    # 访问日志路径+使用的日志格式
    access_log  logs/access.log  main;

    # 开启高效文件传输模式（sendfile系统调用，减少磁盘IO拷贝）
    sendfile        on;
    # 开启TCP_NODELAY选项，提升网络传输效率
    tcp_nodelay     on;
    # 连接超时时间：客户端与Nginx的连接保持时间，单位s，生产环境设60-120
    keepalive_timeout  65;
    # 关闭版本号显示（安全优化，避免泄露Nginx版本）
    server_tokens  off;

    # ================================= server块（虚拟主机，一个server对应一个域名/端口）=================================
    # 示例：监听80端口，域名为localhost/127.0.0.1
    server {
        # 监听端口+IP：listen 80 表示监听所有IP的80端口；listen 192.168.1.100:80 表示仅监听指定IP
        listen       80;
        # 服务器域名/IP，可配置多个，用空格分隔，如server_name localhost 127.0.0.1 www.test.com;
        server_name  localhost;
        # 字符集编码，解决静态资源中文乱码
        charset utf-8;

        # ================================= location块（路径匹配，匹配请求URL的路径部分）=================================
        # 匹配根路径（http://localhost/）
        location / {
            # 静态资源根目录：请求路径对应的本地文件目录
            root   html;
            # 默认索引页，按顺序匹配，存在即返回
            index  index.html index.htm;
            # 开启目录索引（若访问目录无索引页，显示文件列表），生产环境建议关闭
            # autoindex on;
        }

        # 匹配所有以.js结尾的请求，可单独配置缓存、跨域等
        # location ~* \.js$ {
        #     root html;
        #     expires 7d; # 缓存7天
        # }

        # 错误页配置：404错误跳转到/html/404.html，500/502/503/504跳转到50x.html
        error_page   404              /404.html;
        error_page   500 502 503 504  /50x.html;
        # 匹配50x错误页的路径
        location = /50x.html {
            root   html;
        }
    }

    # 可配置多个server块（多虚拟主机），如监听8080端口，或另一个域名
    # server {
    #     listen       8080;
    #     server_name  192.168.1.100;
    #     location / {
    #         root   html8080;
    #         index  index.html;
    #     }
    # }
}

核心块关键参数说明（必记）

  • worker_processes：性能核心参数，直接影响Nginx并发能力，必须根据服务器CPU调整（如4核CPU设为4）；
  • worker_connections：单个进程最大连接数，结合worker_processes可计算最大并发（理论值=worker_processes*worker_connections）；
  • listen：server块核心，指定监听的端口/IP，如listen 443 ssl表示监听443端口并开启HTTPS；
  • server_name：域名匹配，支持精确匹配（www.test.com）、通配符（*.test.com）、正则匹配（~^www\.test\.com$）；
  • root：静态资源根目录，root html表示请求路径对应html目录下的文件，如http://localhost/index.html对应html/index.html；
  • location：路径匹配，是Nginx最灵活的配置，支持**精确匹配（=）、前缀匹配（^~）、正则匹配（~/*）、普通前缀匹配**，匹配优先级从高到低。
配置简化技巧：Linux环境下，主配置nginx.conf可只保留全局、events、http核心参数，将每个server块单独写在conf.d/目录下（如conf.d/test.conf），通过include conf.d/*.conf;引入，便于管理多域名配置。

模块3：实战1 - 静态资源服务（部署Vue/React/纯前端项目）

Nginx处理静态资源（HTML/CSS/JS/图片/视频）的性能远高于Tomcat/Jetty，是**前端项目部署的首选**，也是Java开发中「动静分离」的核心实现（Nginx处理静态资源，Java后端处理动态接口）。

前端项目部署前提：Vue/React项目需先执行npm run build打包，生成dist目录，将dist目录下的所有文件复制到Nginx的静态资源目录即可。

核心配置（部署前端打包后的dist目录，无跨域）

http {
    include       mime.types;
    default_type  application/octet-stream;
    sendfile        on;
    keepalive_timeout  65;
    server_tokens  off;

    # 前端项目部署：监听80端口，域名www.front.com（本地可修改hosts映射）
    server {
        listen       80;
        server_name  www.front.com 127.0.0.1;
        charset utf-8;

        # 静态资源根目录：指向前端打包后的dist目录路径
        # Windows：root D:/nginx-1.24.0/html/front-dist;
        # Linux：root /usr/share/nginx/html/front-dist;
        root   /usr/share/nginx/html/front-dist;
        # 默认索引页：前端打包后的首页，一般为index.html
        index  index.html index.htm;

        # 匹配所有请求路径，处理前端路由（Vue/React路由模式为history时必配）
        # 解决history模式下，刷新页面404的问题
        location / {
            # 若请求的文件/目录不存在，转发到index.html，由前端路由处理
            try_files $uri $uri/ /index.html;
        }

        # 静态资源缓存优化：对CSS/JS/图片/字体设置缓存，提升访问速度
        location ~* \.(css|js|jpg|png|gif|ico|svg|ttf|woff2)$ {
            expires 7d; # 缓存7天
            add_header Cache-Control "public, max-age=604800";
        }

        # 访问日志
        access_log  /var/log/nginx/front-access.log  main;
        error_log   /var/log/nginx/front-error.log   warn;
    }
}

部署步骤+验证方法

  1. 打包前端：执行npm run build，生成dist目录；
  2. 复制文件：将dist目录下的所有文件复制到Nginx静态目录（如Linux/usr/share/nginx/html/front-dist）；
  3. 配置Nginx：将上述配置写入nginx.conf或conf.d/front.conf；
  4. 检测语法：执行nginx -t，提示「nginx: configuration file ... is successful」即语法正确；
  5. 热重载：执行nginx -s reload，配置生效；
  6. 访问验证：浏览器访问http://服务器IP/www.front.com，正常显示前端项目，刷新无404即部署成功。
关键配置：try_files $uri $uri/ /index.html是Vue/React history路由模式的必配项，解决刷新页面404的问题；静态资源缓存配置可大幅提升前端访问速度，生产环境必加。

模块4：实战2 - 反向代理（核心，解决前端跨域/代理Java后端）

**Nginx最常用的功能**，前端开发中解决「浏览器同源策略」导致的跨域问题，Java开发中作为Java后端（SpringBoot/SpringMVC）的前置代理，实现「前端→Nginx→Java后端」的请求转发，是企业开发的标准架构。

反向代理核心原理：客户端请求Nginx，Nginx将请求转发到后端真实服务（如SpringBoot的8080端口），后端处理后将结果返回给Nginx，Nginx再将结果返回给客户端——客户端仅与Nginx交互，无需直接访问后端，从而解决跨域。

实战场景1：前端+后端同域名，解决跨域（推荐，生产环境标准）

配置Nginx，将「静态资源请求（/）」转发到前端目录，「接口请求（/api/）」转发到Java后端（如SpringBoot 8080端口），实现**同域名动静分离+跨域解决**，前端无需配置代理，直接请求/api/xxx即可。

http {
    include       mime.types;
    default_type  application/octet-stream;
    sendfile        on;
    keepalive_timeout  65;
    server_tokens  off;

    # 同域名代理：前端+后端共用www.test.com，通过路径区分
    server {
        listen       80;
        server_name  www.test.com 127.0.0.1;
        charset utf-8;

        # ================= 匹配接口请求 /api/，转发到Java后端（SpringBoot 8080）=================
        location /api/ {
            # 后端真实地址：http://后端IP:后端端口，结尾的/必须加！
            proxy_pass http://127.0.0.1:8080/;
            # 核心：设置反向代理请求头，解决后端获取真实客户端IP、跨域、Session共享问题
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            proxy_set_header X-Forwarded-Proto $scheme;
            # 关闭缓存，实时获取后端结果
            proxy_cache off;
            # 连接超时和读取超时时间
            proxy_connect_timeout 30s;
            proxy_read_timeout 30s;
        }

        # ================= 匹配所有其他请求，转发到前端静态资源目录 =================
        location / {
            root   /usr/share/nginx/html/front-dist; # 前端dist目录
            index  index.html index.htm;
            try_files $uri $uri/ /index.html; # 处理前端history路由
        }

        access_log  /var/log/nginx/test-access.log  main;
        error_log   /var/log/nginx/test-error.log   warn;
    }
}

实战场景2：不同域名，代理后端接口（开发环境临时使用）

前端域名www.front.com，后端代理域名www.api.com，将www.api.com的所有请求转发到Java后端，前端请求http://www.api.com/xxx即可访问后端接口。

http {
    include       mime.types;
    default_type  application/octet-stream;
    sendfile        on;
    keepalive_timeout  65;
    server_tokens  off;

    # 前端域名：部署静态资源
    server {
        listen 80;
        server_name www.front.com;
        root /usr/share/nginx/html/front-dist;
        index index.html;
        try_files $uri $uri/ /index.html;
    }

    # 后端代理域名：所有请求转发到SpringBoot 8080
    server {
        listen 80;
        server_name www.api.com;
        charset utf-8;

        location / {
            proxy_pass http://127.0.0.1:8080/;
            # 必配请求头
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            proxy_connect_timeout 30s;
            proxy_read_timeout 30s;
        }
    }
}

配置验证方法

  1. 启动Java后端：确保SpringBoot服务在8080端口正常运行，接口http://127.0.0.1:8080/hello可访问；
  2. 配置Nginx：写入上述配置，检测语法nginx -t，热重载nginx -s reload；
  3. 前端请求：前端直接请求/api/hello（场景1）或http://www.api.com/hello（场景2）；
  4. 结果验证：前端正常获取后端返回结果，无跨域报错，即配置成功。
反向代理高频坑点：proxy_pass结尾的/必须加对！如后端接口是http://127.0.0.1:8080/api/hello，则proxy_pass http://127.0.0.1:8080/（加/）会转发为http://127.0.0.1:8080/hello，proxy_pass http://127.0.0.1:8080（不加/）会转发为http://127.0.0.1:8080/api/hello，根据实际接口路径调整。

模块5：实战3 - 负载均衡（多Java后端集群，提升并发能力）

当Java后端单服务并发能力不足时，可部署多个后端服务（如8080、8081、8082端口），通过Nginx**负载均衡**将客户端请求均匀分发到各个后端服务，实现**分流减压、故障转移**，是生产环境高可用的核心配置。

Nginx内置4种常用负载均衡策略，无需额外配置，直接选择即可：

  • 轮询（默认）：请求按顺序依次分发到各个后端，简单高效；
  • 权重（weight）：指定后端权重，权重越高，分发的请求越多（如权重2的后端接收的请求是权重1的2倍），适合后端服务器性能不一致的场景；
  • IP哈希（ip_hash）：根据客户端IP进行哈希计算，将同一客户端的所有请求分发到同一个后端，解决**Session共享**问题（无需后端做Session同步）；
  • 最少连接（least_conn）：将请求分发到当前活跃连接数最少的后端，适合请求处理时间不一致的场景。

核心配置（权重策略+IP哈希策略，生产环境常用）

http {
    include       mime.types;
    default_type  application/octet-stream;
    sendfile        on;
    keepalive_timeout  65;
    server_tokens  off;

    # ================================= 定义后端服务集群（upstream 块）=================================
    # 自定义集群名称：backend_cluster，可任意命名
    # 格式：server 后端IP:端口 weight=权重;  weight默认1，max_fails=最大失败次数，fail_timeout=失败超时时间
    upstream backend_cluster {
        # 策略1：权重策略（推荐）—— 8080权重2，8081权重1，8082权重1，请求比例2:1:1
        server 127.0.0.1:8080 weight=2 max_fails=3 fail_timeout=30s;
        server 127.0.0.1:8081 weight=1 max_fails=3 fail_timeout=30s;
        server 127.0.0.1:8082 weight=1 max_fails=3 fail_timeout=30s;

        # 策略2：IP哈希策略（解决Session共享）—— 注释上面的权重配置，打开下面这行
        # ip_hash;
        # server 127.0.0.1:8080;
        # server 127.0.0.1:8081;
        # server 127.0.0.1:8082;

        # 策略3：最少连接策略
        # least_conn;
        # server 127.0.0.1:8080;
        # server 127.0.0.1:8081;
    }

    # ================================= 反向代理到后端集群 =================================
    server {
        listen       80;
        server_name  www.test.com 127.0.0.1;
        charset utf-8;

        # 接口请求转发到后端集群
        location /api/ {
            # proxy_pass 指向集群名称，格式：http://集群名称/
            proxy_pass http://backend_cluster/;
            # 必配反向代理请求头
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            proxy_connect_timeout 30s;
            proxy_read_timeout 30s;
        }

        # 前端静态资源
        location / {
            root   /usr/share/nginx/html/front-dist;
            index  index.html index.htm;
            try_files $uri $uri/ /index.html;
        }
    }
}

部署+验证步骤

  1. 启动多后端服务：将SpringBoot项目打包，分别在8080、8081、8082端口启动，确保所有接口均可正常访问；
  2. 配置Nginx：写入上述配置，检测语法nginx -t，热重载nginx -s reload；
  3. 模拟请求：通过浏览器/Postman多次请求http://www.test.com/api/hello；
  4. 结果验证：查看各后端服务的访问日志，8080端口的请求数约为8081/8082的2倍（权重策略），或同一IP的请求始终分发到同一后端（IP哈希策略），即负载均衡生效。
高可用优化：max_fails=3 fail_timeout=30s表示当某个后端连续3次请求失败时，Nginx会在30秒内不再将请求分发到该后端，实现**故障自动转移**，避免请求到不可用的服务。

模块6：实战4 - HTTPS配置（SSL证书，实现HTTPS/HTTPS跳转）

生产环境要求所有网站使用HTTPS协议（加密传输，保障数据安全），Nginx可通过配置SSL证书实现HTTPS访问，同时可配置**HTTP自动跳转到HTTPS**（强制HTTPS），证书可从阿里云/腾讯云/Let's Encrypt获取（Let's Encrypt为免费证书）。

SSL证书准备：获取证书后，会得到2个核心文件——xxx.crt（公钥文件/证书文件）和xxx.key（私钥文件），将这两个文件上传到Nginx服务器，建议放在单独目录（如Linux/etc/nginx/ssl/，Windowsconf/ssl/）。

核心配置（HTTPS访问 + HTTP自动跳转到HTTPS，生产环境标准）

http {
    include       mime.types;
    default_type  application/octet-stream;
    sendfile        on;
    keepalive_timeout  65;
    server_tokens  off;

    # ================= 配置1：监听80端口，所有HTTP请求自动跳转到HTTPS（443端口）=================
    server {
        listen       80;
        server_name  www.test.com 127.0.0.1;
        # 301永久重定向，跳转到HTTPS地址
        return 301 https://$host$request_uri;
    }

    # ================= 配置2：监听443端口，开启HTTPS，配置SSL证书 =================
    server {
        listen       443 ssl; # 监听443端口，开启SSL
        server_name  www.test.com 127.0.0.1;
        charset utf-8;

        # ================= SSL证书核心配置 =================
        # 公钥/证书文件路径
        ssl_certificate      /etc/nginx/ssl/test.crt;
        # 私钥文件路径
        ssl_certificate_key  /etc/nginx/ssl/test.key;
        # SSL协议版本，禁用低版本（不安全）
        ssl_protocols TLSv1.2 TLSv1.3;
        # 加密套件
        ssl_ciphers ECDHE-RSA-AES128-GCM-SHA256:ECDHE-RSA-AES256-GCM-SHA384;
        # 会话缓存
        ssl_session_cache    shared:SSL:1m;
        # 会话超时时间
        ssl_session_timeout  5m;

        # ================= 业务配置（反向代理/静态资源，和之前一致）=================
        # 接口反向代理
        location /api/ {
            proxy_pass http://backend_cluster/;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            proxy_set_header X-Forwarded-Proto $scheme; # 传递HTTPS协议给后端
        }

        # 前端静态资源
        location / {
            root   /usr/share/nginx/html/front-dist;
            index  index.html index.htm;
            try_files $uri $uri/ /index.html;
        }

        access_log  /var/log/nginx/https-access.log  main;
        error_log   /var/log/nginx/https-error.log   warn;
    }

    # 负载均衡集群配置（若需要）
    upstream backend_cluster {
        server 127.0.0.1:8080 weight=2;
        server 127.0.0.1:8081 weight=1;
    }
}

配置验证方法

  1. 上传证书：将test.crt和test.key上传到配置的证书目录；
  2. 配置Nginx：写入上述配置，检测语法nginx -t，热重载nginx -s reload；
  3. 开放端口：Linux开放443端口（firewall-cmd --add-port=443/tcp --permanent && firewall-cmd --reload）；
  4. 访问验证：浏览器访问http://www.test.com，会自动跳转到https://www.test.com，地址栏显示小锁图标，即HTTPS配置成功。
HTTPS高频坑点：1. 证书文件路径错误（确保ssl_certificate和ssl_certificate_key指向正确的文件）；2. 443端口未开放（防火墙必须开放443端口）；3. 证书域名与配置的server_name不一致（会出现证书不安全提示）。

模块7：Nginx 常用命令（开发/生产必备，熟记）

Nginx所有操作均通过命令行执行，**Windows需进入Nginx解压目录的sbin目录，Linux可直接执行**（已配置环境变量），核心命令分为「启动/停止/重载/检测」四大类，生产环境优先使用「热重载」，避免服务中断。

通用核心命令（Windows+Linux）

# 1. 检测配置语法（最常用，配置修改后必执行）
nginx -t
# 2. 热重载配置（生产环境首选，不中断服务，配置生效）
nginx -s reload
# 3. 启动Nginx
nginx
# 4. 优雅停止Nginx（处理完当前请求后停止，不中断正在处理的请求）
nginx -s stop
# 5. 强制停止Nginx（立即停止，可能中断正在处理的请求，不推荐）
nginx -s quit
# 6. 查看Nginx版本
nginx -v # 简单版本
nginx -V # 详细版本（含编译参数）

Linux 专属命令（systemctl 管理，推荐）

# 1. 启动Nginx
systemctl start nginx
# 2. 停止Nginx
systemctl stop nginx
# 3. 重启Nginx
systemctl restart nginx
# 4. 热重载Nginx
systemctl reload nginx
# 5. 查看Nginx运行状态
systemctl status nginx
# 6. 设置开机自启
systemctl enable nginx
# 7. 关闭开机自启
systemctl disable nginx

模块8：Nginx 新手高频问题排查（避坑必备）

  • 问题1：Nginx启动失败，提示「bind() to 0.0.0.0:80 failed (98: Address already in use)」
    原因：80端口被其他程序占用（如Tomcat、Apache、迅雷）；
    解决：1. 关闭占用80端口的程序（netstat -ano | findstr :80（Windows）/netstat -tulpn | grep :80（Linux）查找PID，然后关闭）；2. 修改Nginx监听端口（如改为8080）。
  • 问题2：配置修改后不生效
    原因：1. 配置语法错误；2. 未执行热重载命令；3. 重载命令执行失败；
    解决：按顺序执行nginx -t（检测语法）→ nginx -s reload（热重载），确保两步都提示成功。
  • 问题3：前端刷新页面404（Vue/React history路由）
    原因：Nginx未配置前端路由重定向，直接访问非首页路径时，Nginx找不到对应的文件；
    解决：在location / 中添加try_files $uri $uri/ /index.html;。
  • 问题4：反向代理后，前端跨域报错「No 'Access-Control-Allow-Origin' header is present on the requested resource」
    原因：1. proxy_set_header Host $host;等核心请求头未配置；2. 后端未配置跨域允许；3. proxy_pass路径错误；
    解决：确保配置了所有核心反向代理请求头，同域名代理可彻底解决跨域。
  • 问题5：浏览器访问Nginx提示「403 Forbidden」
    原因：1. 静态资源目录权限不足（Linux）；2. 根目录配置错误，Nginx找不到文件；3. 目录索引未开启；
    解决：Linux下给静态资源目录赋权（chmod -R 755 /usr/share/nginx/html/），检查root和index配置是否正确。
  • 问题6：负载均衡不生效，所有请求都分发到同一个后端
    原因：1. 仅启动了一个后端服务；2. 配置了ip_hash策略（同一IP会固定分发）；3. upstream集群名称与proxy_pass不一致；
    解决：启动所有后端服务，检查策略配置，确保proxy_pass指向正确的集群名称。
  • 问题7：HTTPS访问提示「您的连接不是私密连接」
    原因：1. 证书过期；2. 证书域名与访问域名不一致；3. 证书文件路径错误；4. 未配置SSL协议版本；
    解决：检查证书有效性和域名匹配性，确保证书路径正确，配置SSLv1.2/1.3协议。

模块9：Nginx 性能优化（生产环境必加）

基础配置可满足开发需求，生产环境需做简单性能优化，提升Nginx并发能力和响应速度，以下配置直接添加到对应块即可。

  • 全局块：worker_processes CPU核心数;（如4核设为4），worker_rlimit_nofile 65535;（提升单个进程最大打开文件数）；
  • events块：worker_connections 65535;（提升单个进程最大连接数），use epoll;（Linux高性能模式）；
  • http块：sendfile on;、tcp_nopush on;、tcp_nodelay on;（提升文件传输和网络效率），keepalive_timeout 60s;（合理设置连接超时）；
  • 静态资源：添加缓存配置（expires 7d;），减少重复请求；
  • 反向代理：添加proxy_cache off;（关闭代理缓存，实时获取后端数据），设置合理的proxy_connect_timeout和proxy_read_timeout。

模块10：Nginx 常用拓展功能（按需添加）

  • 防盗链配置：禁止其他网站引用自己的静态资源（图片/视频），添加到location静态资源块：
    valid_referers www.test.com 127.0.0.1; if ($invalid_referer) { return 403; }
  • 请求限流：限制单IP访问频率，防止恶意攻击，添加到http/server块：
    limit_req_zone $binary_remote_addr zone=one:10m rate=10r/s;（10M内存，单IP每秒10个请求），
    location / { limit_req zone=one; }
  • 压缩传输：开启Gzip压缩，减小传输文件大小，提升访问速度，添加到http块：
    gzip on; gzip_types text/plain text/css application/json application/javascript text/xml;
  • 自定义错误页：替换Nginx默认错误页，更友好，添加到server块：
    error_page 404 /404.html; error_page 500 502 503 504 /50x.html; location = /404.html { root html; }

Nginx 与 Java 开发核心关联总结（重点）

1. 动静分离：Nginx处理静态资源（HTML/CSS/JS），Java后端处理动态接口，提升整体性能；
2. 跨域解决：通过反向代理实现前端与后端同域名，彻底解决浏览器跨域问题，无需后端配置复杂的CORS；
3. 负载均衡：部署多Java后端服务，通过Nginx分流，提升系统并发能力和高可用性；
4. 前置代理：Nginx作为Java后端的前置入口，实现HTTPS配置、请求限流、防盗链、日志统一收集；
5. 服务部署：Nginx可同时部署前端静态资源和代理后端接口，实现「单服务器一站式部署」，简化开发和运维。
